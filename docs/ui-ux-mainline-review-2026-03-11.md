# Cogniforge UI/UX 与交互主线审查报告

日期：2026-03-11

审查范围：
- 以 UI/UX、页面职责、交互主线为主
- 不以代码风格、视觉审美、颜色圆角间距为重点
- 结合前端代码阅读与真实页面走查

实际走查主线：
- 注册 -> 登录 -> Dashboard
- Dashboard -> Problems -> 新建 Problem
- Problems -> ProblemDetail
- ProblemDetail 内完成一次 Socratic 作答
- ProblemDetail 内手动推进步骤
- ProblemDetail 内切换到 Exploration
- 接受派生概念 -> 提升为知识卡 -> 加入复习
- ProblemDetail -> Reviews

## 一、当前产品主线总结

当前产品主线已经比较明确，不是“没有主线”，而是主线已经围绕结构化学习闭环搭起来了：

1. `Dashboard` 承担入口与优先级分发
   - 告诉用户先继续问题，还是先处理复习
   - 提供进入 Problems / Reviews / Model Cards 的快捷入口

2. `Problems` 是学习问题库，也是进入学习工作区的主入口
   - 用户从这里创建问题
   - 再进入单个问题的 `ProblemDetail`

3. `ProblemDetail` 是真正的核心工作区
   - 展示当前问题、当前学习路径、当前步骤
   - 在这里切换 `socratic` / `exploration`
   - 在这里产生结构化学习产物：掌握度、派生概念、派生路径、复习线索

4. `Model Cards` 是长期知识资产库
   - 承接 ProblemDetail 中被确认与沉淀的知识
   - 继续编辑、修订、排入 recall

5. `Reviews` 是复习生命周期总览页
   - 承担 recall 队列、知识卡演化状态、阶段性复盘归档

6. `SRS Review` 是 recall 执行子流程
   - 负责单次主动回忆，而不是复习总控页

总结判断：
- 主线存在，而且符合“问题驱动学习 -> 工作区产出结构化结果 -> 沉淀知识卡 -> 进入复习”的产品方向
- 真正的问题不在“有没有主线”，而在于：
  - 主线在 `ProblemDetail` 里过于拥挤
  - 模式边界、回合边界、步骤推进边界不够干净
  - 若请求失败，很多页面会伪装成“没有数据”或“已经完成”

## 二、问题清单

### P0-1：Socratic 主协议可被绕开，且绕开后页面会进入自相矛盾状态

- 位置：
  - `ProblemDetail` 当前步骤区
  - 手动步骤推进逻辑
- 现象：
  - 在 `老师提问我来答` 模式下，用户仍可直接点击“标记当前步骤完成”
  - 这会在没有完成评估闭环的情况下把步骤推进到下一步
  - 推进后，页面会同时出现：
    - 当前步骤已经变成下一步
    - 但上一轮提示、上一轮问题语境、上一轮产物仍然残留
- 为什么影响体验：
  - 这直接破坏了 Socratic 模式的核心承诺：系统评估后再决定是否推进
  - 用户会不知道自己现在到底是“被评估推进了”，还是“自己手动跳步了”
  - 一旦页面同时显示“第 2 步”与“第 1 步遗留提示”，用户对路径与当前任务的信任会迅速下降
- 建议修改：
  - 在 `socratic` 模式下默认取消手动“标记当前步骤完成”
  - 如果必须保留，改成显式 override，并增加确认说明
  - 任意步骤切换后，必须清空或重建与步骤绑定的提示、问题、当前回合摘要，避免旧内容残留

### P0-2：关键页面失败时会伪装成“空状态”或“已完成”，用户无法判断是没数据还是加载失败

- 位置：
  - `Dashboard`
  - `Problems`
  - `Reviews`
  - `ProblemDetail`
  - `SRS Review`
- 现象：
  - 多个页面在请求失败时只写控制台日志，没有用户可见错误反馈
  - `Problems`/`Reviews`/`Dashboard` 失败后很容易显示成“没有数据”
  - `SRS Review` 在加载失败时会直接落到“今日复习已全部完成”
- 为什么影响体验：
  - 这会直接破坏“我现在该做什么”的判断
  - 用户可能以为今天没有待复习、没有问题、没有知识卡，实际上只是请求失败
  - 对主线产品来说，这比普通报错更危险，因为它会误导用户停止行动
- 建议修改：
  - 所有主线页面区分三类状态：`loading / empty / error`
  - `error` 必须有明确文案与重试按钮
  - 如果已有旧数据，优先保留旧数据并显示“刷新失败”，不要直接退化成空页面
  - `SRS Review` 必须把“加载失败”与“今日已完成”严格分离

### P1-1：问题创建入口没有要求用户选择当前学习协议，导致第一步任务不够明确

- 位置：
  - `Problems` 页创建问题弹窗
- 现象：
  - 创建问题时只填写标题、描述、概念
  - 不让用户声明是想进入 `老师提问我来答` 还是 `我来提问探索概念`
  - 创建完成后直接进入默认 Socratic
- 为什么影响体验：
  - `learning_mode` 是产品的一等概念，但入口没有体现
  - 对第一次进入工作区的用户来说，“现在我应该回答老师，还是主动提问”其实是最关键的任务判断
  - 现在这个判断被延后到了 ProblemDetail 内部，增加了一次心智切换
- 建议修改：
  - 在创建问题时增加“从哪种学习方式开始”
  - 或在创建成功后先给一个轻量的进入选择页/底部行动条
  - 让用户带着明确意图进入 ProblemDetail，而不是进入后再改模式

### P1-2：ProblemDetail 过度堆叠同类摘要，导致“信息都在，但主次不清”

- 位置：
  - `ProblemDetail` 头部
  - 工作区总览卡
  - 状态条
  - 当前步骤卡
  - 右侧本轮产物区
- 现象：
  - 当前步骤、模式、进度、下一步动作、本轮产出等信息被重复放在多个卡片中
  - 同时右侧又承载概念治理、路径治理、知识卡提升、加入复习等大量动作
- 为什么影响体验：
  - 这类页面的问题不是“信息不够”，而是“优先级不够”
  - 用户一进入页面就同时面对：
    - 继续当前步骤
    - 切模式
    - 看产物
    - 审核概念
    - 决定路径
    - 打开知识卡
    - 打开复习中心
  - 结果是当前学习动作和产物治理动作互相抢焦点
- 建议修改：
  - 只保留一个首屏主导区：当前步骤 + 当前模式 + 一个明确下一步动作
  - 把“本轮产物”改成回合结束后的二级阶段，而不是和主输入区同级抢注意力
  - 概念/路径治理默认折叠成摘要，进入“处理本轮产物”后再展开批量处理

### P1-3：模式切换没有形成干净的“新回合边界”，上一模式的产物会污染当前模式

- 位置：
  - `ProblemDetail` 的工作区摘要与右侧产物区
- 现象：
  - 从 Socratic 切到 Exploration 后，页面文案已经告诉用户“你现在在探索回合”
  - 但“本轮产出”仍可能继续显示上一轮 Socratic 的掌握度、概念候选、路径候选
  - 当前模式还没开始，页面却在展示上一模式遗留产物
- 为什么影响体验：
  - 用户无法判断当前看到的是：
    - 当前模式的当前回合
    - 还是上一回合的历史残留
  - 这会直接伤害“我现在在哪、这是谁的结果、下一步到底看哪块”
- 建议修改：
  - 模式切换时建立清晰的回合分割
  - 当前模式未开始时，明确显示“当前模式尚无产出”
  - 把上一轮内容单独归为“上一轮产出”或“最近一次产出”，不要继续占据“本轮产出”位置

### P1-4：Reviews 页职责过多，而且知识卡生命周期卡片信息密度过高，不利于快速决策

- 位置：
  - `Reviews` 页
- 现象：
  - 同一页同时承担：
    - 到期 recall 队列
    - 知识卡生命周期
    - 阶段性复盘归档
    - 手动新建复盘
  - 生命周期卡片把来源、模式、原始回合摘录等拼成一大段文案
- 为什么影响体验：
  - 用户进入 Reviews 页最需要的是先回答：“先做 recall，还是先修卡，还是先看复盘”
  - 现在这三个目标并列存在，但层级不够强
  - 长段落描述削弱了扫描效率，用户不容易快速选定下一步
- 建议修改：
  - 强化首要决策顺序：
    - 有到期 recall：先 recall
    - 无到期 recall：再看哪些知识卡要修
    - 复盘归档降级为次级区块或折叠区
  - 生命周期卡片改为结构化三行：
    - 来源
    - 当前状态
    - 建议动作
  - 不要把长来源摘录直接作为列表主文案

### P2-1：Dashboard 是一个合理的入口分发页，但“学习”导航名仍然略偏泛，首次进入时不够像行动入口

- 位置：
  - 全局导航
  - `Dashboard`
- 现象：
  - 一级导航把 Dashboard 命名为“学习”
  - 但这个页面本质上是一个总览/分发页，而不是实际学习工作区
- 为什么影响体验：
  - 熟悉后问题不大，但第一次使用时，用户会以为“学习”就是直接开始学
  - 实际真正开始学是在 `Problems -> ProblemDetail`
- 建议修改：
  - 二选一即可：
    - 把导航名改成“总览”或“继续学习”
    - 或保留“学习”，但让首页主卡更强地表现为“立即继续这个问题 / 立即开始复习”

### P2-2：ProblemDetail 里的产物治理动作太早暴露，容易把“学习工作区”变成“审核后台”

- 位置：
  - `ProblemDetail` 右侧本轮产物区
- 现象：
  - 一轮学习结束后，用户很快就会看到：
    - 接受 / 驳回 / 暂缓
    - 合并
    - 提升为知识卡
    - 加入复习
    - 路径插入主线 / 保存为支线 / 稍后再看 / 忽略
  - 这些动作本身合理，但在主学习界面里集中出现后，容易像治理后台
- 为什么影响体验：
  - 用户原本的主任务是继续学习
  - 但页面很快把注意力拉去做概念审核、路径编排和资产沉淀
  - 这会让“学习”与“治理”在同一层级争夺注意力
- 建议修改：
  - 采用分阶段披露：
    - 默认只显示产物摘要和推荐动作
    - 点击“处理本轮产物”后，再展开治理动作
  - 把概念治理和路径治理设计成明确的二级工作流，而不是默认展开的并行工作

## 三、优先改造建议

建议优先级：

1. 先修协议层冲突
   - 收掉或约束 Socratic 下的手动跳步
   - 处理步骤切换后的旧提示/旧产物残留

2. 再修状态反馈
   - 主线页面补全 `loading / empty / error`
   - 让失败不再伪装成“没数据”或“已完成”

3. 再修 ProblemDetail 的信息层级
   - 压缩重复摘要
   - 让当前动作永远压过产物治理动作

4. 最后修跨页职责清晰度
   - 问题创建时前置学习模式
   - Reviews 页突出“先 recall 还是先修卡”

## 四、结论

这个项目的主线方向是对的，核心工作区也已经落在 `ProblemDetail` 上了。当前最主要的问题不是缺少功能，而是主线在关键节点上还不够“干净”：

- Socratic 协议会被手动推进打断
- 模式切换后的回合边界不够清楚
- ProblemDetail 里学习动作与治理动作抢焦点
- 请求失败时，多个主线页面会让用户误判“没有任务”

如果优先把这四类问题收紧，产品会明显更接近“结构化学习系统”，而不是“功能很多但当前任务不够稳定的学习面板”。
