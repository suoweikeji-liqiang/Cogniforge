## 当前阶段判断

主线功能已从“补骨架”进入“收尾和生产化”阶段。接下来不应再把精力平均摊到所有模块，而是收敛到以下目标：

1. 把验收口径和文档更新到当前事实。
2. 把安全与会话状态从内存态推进到持久化。
3. 给轻量 RAG 增加最小可观测性。
4. 建立前端自动化回归基线。

截至 2026-03-04，本轮 1-4 已完成第一阶段闭环:

- 文档已回写到当前真实状态。
- token blacklist 与登录限流已持久化到数据库。
- RAG 已有 retrieval log / summary 接口，并接到 Dashboard 观察卡。
- 前端已补一条浏览器级 Dashboard smoke，作为回归基线。

## 目标

把当前仓库从“功能 beta”推进到“可验收、可维护、可继续扩展”的版本，而不是继续堆新功能。

## 总体策略

1. 不再大规模新增业务模块。
2. 优先做生产化短板，而不是继续扩张功能面。
3. 尽量让每一项改动都带测试或验证脚本。

## 阶段规划

### 阶段 1: 文档与验收同步

状态: 已完成

目标: 让文档、代码、测试结论一致。

- 更新 SRS gap 分析文档到当前状态。
- 更新收尾计划，聚焦剩余高优先级项。
- 整理一份当前验证基线:
  - 前端 build
  - 后端 smoke tests
  - PostgreSQL/pgvector 验证

### 阶段 2: 安全与会话持久化

状态: 已完成

目标: 去掉当前明显的内存态风险。

- 为 token blacklist 增加持久化存储。
- 为登录限流增加持久化存储。
- 确保服务重启后 refresh/logout/限流状态不丢失。
- 补相关回归测试。

### 阶段 3: RAG 可观测性

状态: 已完成第一阶段

目标: 让检索增强不仅能跑，还能排查和评估。

- 为 `build_retrieval_context()` 增加召回日志记录。
- 记录:
  - query
  - 命中的对象类型
  - 命中数量
  - Top items
  - 生成请求对应场景（problem/practice/challenge/chat）
- 后台增加最小查询接口或调试页。
- Dashboard 增加最小观察入口。

### 阶段 4: 前端自动化回归

状态: 已完成第一阶段

目标: 不再只靠 build 发现 UI 回归。

- 已建立登录 -> Dashboard 浏览器 smoke。
- 下一步扩展到:
  - 问题创建
  - Model Card 创建
  - Review 生成
  - SRS 入口
  - 管理端 LLM 配置
- 继续保持先做 happy path，再逐步补异常路径。

## 实施优先级

### 当前 P0

- 前端 smoke 从 Dashboard 扩到更多关键流程
- 文档与 README 补充新的验证命令与召回观测入口

### 当前 P1

- RAG 命中质量评估与更完整调试面板
- 管理端和学习主流程的更多前端 smoke

### P2

- 迁移尝试记录闭环
- 更完整的性能与调用指标

## 风险与取舍

### 高风险

- 会话与限流持久化会触及鉴权边界，必须带测试。

### 中风险

- RAG 日志如果设计过重，会污染主流程；应先做最小 schema。

### 低风险

- 文档同步
- 前端 smoke 基线

## 推荐推进顺序

1. 更新文档
2. 持久化 auth 状态
3. 为检索增强加日志
4. 补前端 smoke/e2e

## 验收标准

- 文档不再与代码状态明显冲突
- 服务重启后 refresh/logout/限流状态仍可追踪
- RAG 请求能看到基本召回记录
- 前端至少有一条可跑通的 happy-path 自动化流程
