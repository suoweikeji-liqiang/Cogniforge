# Cogniforge UI 交互审核报告

**审核日期**: 2026-03-11
**审核范围**: 前端交互设计、用户任务流程、信息架构
**审核标准**: 主线清晰度、任务完成效率、认知负荷、系统反馈质量

---

## 执行摘要

**核心问题**: Cogniforge 当前的 UI 是围绕**数据结构**而非**用户任务**设计的。系统提供了 14+ 个顶级页面，但缺乏清晰的主线任务流程。用户面对的是一个功能丰富但方向模糊的工具集合，而不是一个引导他们完成学习目标的系统。

**严重程度**: 🔴 高 - 需要流程重构，而非视觉优化

---

## 一、路由架构问题

### 当前状态
系统有 **14 个主要路由**（不含管理后台）：
1. `/dashboard` - 仪表板
2. `/problems` - 问题列表
3. `/problems/:id` - 问题详情（工作区）
4. `/model-cards` - 模型卡列表
5. `/model-cards/:id` - 模型卡详情
6. `/practice` - 练习
7. `/reviews` - 复习
8. `/srs-review` - SRS 复习会话
9. `/chat` - 对话
10. `/knowledge-graph` - 知识图谱
11. `/challenges` - 挑战
12. `/resources` - 资源
13. `/notes` - 笔记
14. `/cog-test` - 认知测试

### 问题诊断

**问题 1: 路由扁平化导致职责不清**
- 所有功能都是顶级路由，没有层级关系
- 用户无法从 URL 结构理解功能之间的关系
- `/reviews` 和 `/srs-review` 是什么关系？为什么分开？
- `/practice` 和 `/problems` 的区别是什么？

**问题 2: 功能重复和概念混淆**
- **Reviews vs SRS-Review**: 两个复习页面，职责重叠
  - `/reviews` 显示复习队列、模型卡、历史复习记录
  - `/srs-review` 是实际的复习会话界面
  - 用户需要在两个页面之间跳转才能完成一次复习

- **Problems vs Practice**:
  - Problems 是"学习工作区"
  - Practice 是"练习任务"
  - 但从命名和页面内容看，边界模糊

- **Chat vs Problem Workspace**:
  - Chat 页面有 "Legacy" 警告，提示用户去 Problems
  - 说明 Chat 已被 Problem Workspace 取代
  - 但为什么还保留？用户会困惑

**问题 3: 次要功能占据主导航**
- `/knowledge-graph`, `/challenges`, `/resources`, `/notes`, `/cog-test` 都是顶级路由
- 这些功能在核心学习流程中的位置不明确
- 用户不知道什么时候应该使用这些功能

## 二、首页（Dashboard）分析

### 当前设计
Dashboard 包含：
1. **Focus Card** - 动态推荐下一步行动（复习/继续问题/创建问题）
2. **三个指标卡** - 活跃问题数、待复习数、模型卡数
3. **最近问题列表** - 最多 5 个
4. **复习队列** - 最多 4 个待复习卡片
5. **最近模型卡** - 最多 4 个
6. **快速操作区** - 4 个操作卡片（新问题、探索、模型库、复习队列）

### 问题诊断

**✅ 做得好的地方**:
- Focus Card 提供了动态的主要行动建议
- 有优先级逻辑（复习 > 继续问题 > 创建问题）
- 指标卡提供了系统状态概览

**❌ 严重问题**:

**问题 1: 信息过载**
- 一个页面包含 6 个不同的内容区域
- 用户需要扫描整个页面才能决定下一步做什么
- Focus Card 已经给出建议，但下方还有 4 个"快速操作"，稀释了焦点

**问题 2: 重复信息**
- Focus Card 推荐"继续最近的问题"
- 下方"最近问题"列表显示相同的问题
- "快速操作"区又有"探索工作区"指向同一个问题
- 同一个行动出现 3 次，用户不知道点哪个

**问题 3: 缺乏任务阶段引导**
- Dashboard 展示了所有数据，但没有告诉用户：
  - 我现在处于学习流程的哪个阶段？
  - 我应该先做什么，后做什么？
  - 完成当前任务后，下一步是什么？

**问题 4: "快速操作"区职责不清**
- 4 个操作卡片：新问题、探索、模型库、复习队列
- 这些不是"快速操作"，而是"导航入口"
- 与顶部导航栏功能重复
- 用户不知道为什么这 4 个功能被特别强调

## 三、核心流程分析：Problem Workspace

### 当前设计
Problem Detail 页面（`/problems/:id`）是系统最复杂的页面（76.6KB），包含：
- 问题标题、描述、状态
- 学习模式切换（Exploration / Socratic）
- 进度显示
- 工作区概览卡片
- 学习路径管理
- 对话交互区
- 概念候选管理
- 关联概念显示

### 问题诊断

**问题 1: 页面职责过重**
- 这个页面试图同时完成：
  1. 展示问题信息
  2. 管理学习路径
  3. 进行对话交互
  4. 提取和管理概念
  5. 关联模型卡
- 用户在一个页面内需要处理 5 种不同的任务
- 没有清晰的任务阶段划分

**问题 2: 学习模式切换的意义不明确**
- 页面顶部有"Exploration"和"Socratic"模式切换
- 但从代码看，这只是改变了一些文案和 AI 行为
- 用户不知道：
  - 什么时候应该用哪个模式？
  - 切换模式会发生什么？
  - 两种模式的区别是什么？

**问题 3: 概念管理流程断裂**
- 系统会从对话中提取"概念候选"
- 用户需要批准概念候选，才能转为"关联概念"
- 关联概念可以创建模型卡
- 但这个流程在 UI 中是分散的：
  - 概念候选在一个区域
  - 关联概念在另一个区域
  - 创建模型卡的按钮又在别处
- 用户不理解这个流程的目的和价值

**问题 4: 缺乏进度感和成就感**
- 虽然有"进度：X/Y"显示
- 但用户不知道：
  - 完成所有步骤后会发生什么？
  - 我离"解决这个问题"还有多远？
  - 我的学习成果在哪里？

## 四、复习系统分析

### 当前设计
复习功能分散在 3 个页面：
1. **`/reviews`** - 复习中心
   - 显示待复习队列
   - 显示模型卡列表
   - 显示历史复习记录
   - 可以创建新的"Review"（日/周/月总结）

2. **`/srs-review`** - SRS 复习会话
   - 实际的卡片复习界面
   - 显示卡片内容
   - 评分（0-5）
   - 显示下次复习时间

3. **`/model-cards`** - 模型卡库
   - 模型卡列表和筛选
   - 可以添加到复习计划
   - 显示复习状态

### 问题诊断

**问题 1: 流程分裂**
用户要完成一次复习，需要：
1. 去 `/reviews` 看有哪些待复习
2. 点击"开始复习"跳转到 `/srs-review`
3. 在 `/srs-review` 完成复习
4. 如果需要回到工作区强化，跳转到 `/problems/:id`
5. 如果想查看模型卡详情，跳转到 `/model-cards/:id`

这是 **5 个页面** 的跳转，才能完成一个完整的复习循环。

**问题 2: 概念混淆**
- "Reviews" 页面同时显示：
  - SRS 复习队列（基于间隔重复算法）
  - 手动创建的"Review"（学习总结）
- 这是两个完全不同的概念，但放在同一个页面
- 用户会困惑："Review" 到底是指什么？

**问题 3: 模型卡的位置尴尬**
- 模型卡是复习系统的核心
- 但 `/model-cards` 页面看起来像一个"库"或"档案"
- 用户不理解：
  - 模型卡是如何产生的？（从 Problem 中提取概念）
  - 为什么要复习模型卡？（强化记忆）
  - 模型卡和 Problem 的关系是什么？

**问题 4: 复习反馈不足**
- 复习完成后，只显示"下次复习时间"
- 没有告诉用户：
  - 我今天的复习表现如何？
  - 我的整体记忆状态如何？
  - 哪些概念需要重点关注？

## 五、次要功能分析

### Practice 页面
**问题**:
- 页面顶部有"Secondary Surface Banner"提示这是次要功能
- 但它仍然是顶级路由
- 页面内容稀少（任务列表 + 提交历史）
- 与 Problem Workspace 的关系不清楚

**建议**: 应该降级为 Problem 的子功能，或者完全移除

### Chat 页面
**问题**:
- 页面有明确的"Legacy"警告
- 提示用户去 Problems 页面
- 但仍然保留在导航中

**建议**: 应该移除，或者作为隐藏功能保留（不在主导航显示）

### Knowledge Graph, Challenges, Resources, Notes, Cog Test
**问题**:
- 这些页面在代码中存在，但我没有看到实现细节
- 它们都是顶级路由
- 用户不知道什么时候应该使用这些功能
- 它们与核心学习流程的关系不明确

- 要么整合到核心流程中（如 Knowledge Graph 可以是 Model Cards 的可视化视图）

## 六、信息架构问题总结

### 当前架构的本质问题

**围绕数据结构设计，而非用户任务**:
- 有 Problems 表 → 做一个 Problems 页面
- 有 Model Cards 表 → 做一个 Model Cards 页面
- 有 Reviews 表 → 做一个 Reviews 页面
- 有 Practice Tasks 表 → 做一个 Practice 页面

这是典型的"数据库驱动设计"，而不是"任务驱动设计"。

### 用户真正的任务流程应该是

**阶段 1: 定义学习目标**
- 创建一个 Problem（学习挑战）
- 描述我想理解什么

**阶段 2: 探索和理解**
- 在 Problem Workspace 中与 AI 对话
- 系统引导我通过学习路径深入理解
- 系统提取关键概念

**阶段 3: 构建认知模型**
- 将提取的概念转化为 Model Cards
- 添加例子、反例、笔记
- 建立概念之间的联系

**阶段 4: 巩固和内化**
- 通过 SRS 复习强化记忆
- 系统识别薄弱环节
- 引导我回到 Workspace 重新学习

**阶段 5: 应用和迁移**
- 在新的 Problem 中应用已学概念
- 跨领域迁移
- 测试理解深度

### 当前 UI 的问题

**用户看到的是**:
- 14 个平行的功能入口
- 没有明确的起点和终点
- 不知道应该按什么顺序使用这些功能
- 不知道自己处于学习流程的哪个阶段

**用户应该看到的是**:
- 一个清晰的主线任务流程
- 每个阶段的明确目标
- 完成当前阶段后，自动引导到下一阶段
- 支线功能在需要时才出现，不干扰主线

## 七、具体重构建议

### 建议 1: 重新组织路由层级

**主线流程**（顶级导航）:
```
/workspace/:id  （原 /problems/:id）
  - 这是核心工作区，用户 80% 的时间在这里
  - 整合对话、学习路径、概念提取

/library  （原 /model-cards）
  - 我的认知模型库
  - 整合复习队列、模型卡管理

/review  （合并 /reviews 和 /srs-review）
  - 统一的复习界面
  - 根据状态显示不同视图（队列/会话/历史）
```

**支线功能**（二级导航或工具栏）:
```
/workspaces  （原 /problems）
  - 所有工作区的列表
  - 从 Dashboard 或 Library 进入，不是主导航

/insights
  - 整合 Knowledge Graph, 学习统计
  - 作为"查看进度"的入口
```

**移除或降级**:
- `/practice` → 整合到 Workspace 中作为"练习模式"
- `/chat` → 移除（已被 Workspace 取代）
- `/challenges`, `/resources`, `/notes` → 评估是否真的需要，或作为实验性功能隐藏
- `/cog-test` → 如果不是核心功能，移到设置或工具区

### 建议 2: 重新设计 Dashboard

**当前问题**: 信息过载，缺乏焦点

**重构方案**:

```
[Dashboard 新结构]

1. 主要行动区（占据 60% 视觉权重）
   - 只显示 1 个主要行动
   - 大按钮，清晰的文案
   - 例如："继续学习：控制理论基础"

2. 次要行动（占据 20% 视觉权重）
   - 最多 2 个次要行动
   - 例如："3 个概念待复习" + "查看学习统计"

3. 快速访问（占据 20% 视觉权重）
   - 最近的 3 个工作区（缩略卡片）
   - "查看所有工作区"链接
```

**关键原则**:
- 一次只推荐 1 个主要行动
- 移除所有重复信息
- 移除"快速操作"区（与导航重复）
- 用户应该在 3 秒内知道"我现在应该做什么"

### 建议 3: 拆分 Problem Workspace

**当前问题**: 一个页面承担太多职责

**重构方案**:

将 Problem Workspace 拆分为 **3 个标签页**（Tab）:

**Tab 1: 探索（Explore）**
- 对话界面
- 学习路径导航
- 这是用户的主要工作区

**Tab 2: 概念（Concepts）**
- 概念候选列表
- 关联概念列表
- 创建/编辑模型卡
- 这是"整理和构建"的区域

**Tab 3: 总结（Summary）**
- 问题描述和元信息
- 学习进度和统计
- 相关资源链接
- 这是"回顾和导航"的区域

**关键原则**:
- 每个标签页只做一件事
- 默认打开"探索"标签（主要任务）
- 用户可以随时切换，但不会被其他信息干扰

### 建议 4: 统一复习流程

**当前问题**: 复习流程分散在 3 个页面

**重构方案**:

**单一复习入口** (`/review`):

**状态 1: 有待复习卡片**
- 直接显示复习界面（原 /srs-review 的内容）
- 顶部显示进度："3/10 已完成"
- 完成后自动加载下一张卡片

**状态 2: 无待复习卡片**
- 显示复习队列管理
- 显示下次复习时间
- 提供"浏览模型库"入口

**状态 3: 复习完成**
- 显示今日复习总结
- 显示需要强化的概念
- 提供"回到工作区强化"的直接链接

**关键原则**:
- 用户不需要在多个页面之间跳转
- 系统根据状态自动切换视图
- 复习完成后，明确告诉用户下一步做什么

### 建议 5: 明确模型卡的定位

**当前问题**: 模型卡看起来像一个独立的"库"，与学习流程脱节

**重构方案**:

**重命名**: `/model-cards` → `/library`（我的认知库）

**页面结构**:
```
[Library 页面]

1. 顶部状态栏
   - "3 个概念需要复习"（高亮显示）
   - "12 个概念正在学习"
   - "5 个概念已掌握"

2. 主要视图（默认）
   - 按"需要关注"排序的模型卡
   - 每张卡片显示：
     * 概念名称
     * 来源工作区（点击可回到 Workspace）
     * 复习状态和下次复习时间
     * 快速操作：复习/编辑/查看详情

3. 筛选和视图切换
   - 按状态筛选（需要复习/学习中/已掌握）
   - 按来源筛选（按工作区）
   - 切换视图（列表/卡片/知识图谱）
```

**关键原则**:
- 强调模型卡与工作区的关系
- 突出"需要行动"的卡片
- 提供快速回到工作区的路径

## 八、系统反馈问题

### 当前状态

**缺乏任务完成反馈**:
- 用户完成一个学习路径步骤后，没有明确的"完成"提示
- 创建模型卡后，没有告诉用户"下一步做什么"
- 复习完成后，只显示下次复习时间，没有成就感

**缺乏进度可视化**:
- 虽然有"进度：X/Y"，但这只是数字
- 用户不知道自己离目标还有多远
- 没有"里程碑"或"阶段完成"的概念

**缺乏错误恢复指导**:
- 如果用户不知道下一步做什么，系统没有提供帮助
- 如果用户迷路了，没有"回到主线"的明确路径

### 重构建议

**1. 任务完成反馈**
- 每完成一个步骤，显示明确的"✓ 已完成"提示
- 自动推荐下一步行动
- 提供"继续"和"稍后再做"两个选项

**2. 进度可视化**
- 使用进度条而不是"X/Y"数字
- 定义"里程碑"（如"完成 50% 的学习路径"）
- 显示预计完成时间

**3. 迷路恢复**
- 每个页面都有"回到 Dashboard"的明显入口
- Dashboard 始终显示"你现在应该做什么"
- 提供"我不知道该做什么"的帮助入口

## 九、最终判断

### 这个界面是否能让用户顺畅完成核心任务？

**❌ 不能**

用户面对的是一个功能丰富但方向模糊的系统。他们需要自己理解：
- 14 个页面分别是做什么的
- 这些页面之间的关系是什么
- 应该按什么顺序使用这些功能
- 自己现在处于学习流程的哪个阶段

这不是"学习系统"，而是"学习工具箱"。用户需要先学会如何使用这个系统，才能用它来学习其他知识。

### 主路径是否被支线打断？

**❌ 是的**

系统没有明确的"主路径"。所有功能都是平等的顶级路由，用户不知道哪个是主线，哪个是支线。

即使是核心流程（Problem → Concept → Model Card → Review），也被分散在 4 个独立的页面中，用户需要自己记住这个流程。

### 页面职责是否清晰？

**❌ 不清晰**

- Dashboard：既是"推荐下一步"，又是"数据概览"，又是"快速导航"
- Reviews：既是"SRS 复习队列"，又是"学习总结归档"
- Problem Workspace：既是"对话界面"，又是"概念管理"，又是"学习路径导航"

每个页面都试图做太多事情。

### 信息是否按任务阶段组织？

**❌ 不是**

信息是按**数据类型**组织的（Problems, Model Cards, Reviews），而不是按**任务阶段**组织的（探索, 构建, 巩固, 应用）。

### 用户是否始终知道自己在哪、能做什么、下一步是什么？

**❌ 不知道**

- 用户不知道自己在学习流程的哪个阶段
- 用户不知道完成当前任务后应该做什么
- 用户不知道为什么要做某个操作（如"为什么要创建模型卡？"）

### 系统反馈是否让用户安心，而不是困惑？

**❌ 困惑**

- 缺乏任务完成的明确反馈
- 缺乏进度可视化
- 缺乏"下一步"的明确指引
- 用户经常处于"我现在应该做什么？"的状态

## 十、优先级建议

### P0（立即执行）- 流程重构

1. **合并 Reviews 和 SRS-Review**
   - 统一复习流程，减少页面跳转
   - 影响：大幅提升复习体验

2. **重新设计 Dashboard**
   - 移除重复信息
   - 明确"下一步行动"
   - 影响：用户打开系统后立即知道该做什么

3. **移除或降级次要功能**
   - 移除 Chat（已有 Legacy 警告）
   - 评估 Practice, Challenges, Resources, Notes 的必要性
   - 影响：减少认知负荷，聚焦核心流程

### P1（近期执行）- 职责拆分

4. **拆分 Problem Workspace**
   - 使用标签页分离"探索"、"概念"、"总结"
   - 影响：降低单页面复杂度

5. **重新定位 Model Cards**
   - 重命名为"Library"
   - 强调与 Workspace 的关系
   - 影响：用户理解模型卡的价值

### P2（中期执行）- 体验优化

6. **增强系统反馈**
   - 任务完成提示
   - 进度可视化
   - 下一步推荐

7. **优化信息架构**
   - 重新组织路由层级
   - 建立主线/支线的清晰区分

## 结论

Cogniforge 的核心问题不是视觉设计，而是**信息架构和任务流程设计**。

当前系统是围绕数据结构构建的，导致：
- 功能分散，缺乏主线
- 页面职责不清
- 用户需要自己理解系统逻辑

需要的不是"优化 UI"，而是**重新设计用户旅程**：
1. 明确核心任务流程（探索 → 构建 → 巩固 → 应用）
2. 围绕任务阶段重组页面
3. 建立清晰的主线/支线区分
4. 增强系统反馈和引导

这是一个**流程重构项目**，而不是视觉优化项目。
